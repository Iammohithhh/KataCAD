## 2. What KatACAD is, in one paragraph

KatACAD is a parametric, history-based, B-Rep CAD system. A user describes a part in plain language. The system produces a real engineering model that a machine shop can manufacture from the resulting STEP file. Underneath, every model is represented as a feature tree written in a small domain-specific language. Every face, edge, and feature in the tree carries a human-readable name (the "semantic naming layer"). Before any model reaches the user, a deterministic verification engine checks that it is geometrically valid, physically reasonable, and matches what the user asked for.

The product is a multi-quarter build. **This repository contains the two-day exhibition demo of that product**, not the product itself. Keep that distinction sharp.

## 3. What we are building in this repository

The KatACAD **exhibition demo**: a single web application that runs on a booth laptop at a trade show. A visitor types a sentence in plain English. Three seconds later, a fully formed mechanical assembly appears in a cinematic 3D viewport, lit beautifully, often moving. They click a button and see four named verification checks turn green. They click another button and see an A3 technical dossier with orthographic engineering views, a dimension table, a bill of materials, a material specification, and a verification certificate. They click a third button and download a real STEP file that opens cleanly in SolidWorks.

The whole sequence takes under 60 seconds. The demo runs for eight hours a day for the length of the exhibition.

The full specification for what the demo will and will not do lives in `docs/TEAM_REFERENCE.pdf`. Anything the demo "will not do" listed in that document **must not be added** without an explicit instruction from the user.

## 4. Architecture: the three-layer system

The visitor sees one input box, one viewport, one feature tree, one dossier, one export button. Behind the box sit three layers. A router decides which layer answers each prompt.

- **Layer 1 — Hero assemblies.** Eight hand-built parametric flagship pieces (planetary gearbox, five-finger gripper, six-axis robot arm, quadcopter, V-twin engine, differential, bicycle, NEMA mount). Each has real parameter control and animated mechanisms. These carry the booth visually.
- **Layer 2 — Programmatic families.** Ten archetypes generated by CadQuery scripts (bracket, flange, plate, shaft, gear, housing, pulley, hub, manifold, clamp). Real geometry, real parameter control. Cover the long tail of common engineering parts.
- **Layer 3 — Curated dataset retrieval.** Around 500 parts curated from the GenCAD-Code dataset (HuggingFace: `CADCODER/GenCAD-Code`). Indexed in FAISS by their captions. Retrieved parts are executed as CadQuery code through the same Layer 2 pipeline, so they live in the same lighting, with the same exports, with the same feature tree.

The **router** is an OpenAI API call with a system prompt that classifies the user's prompt into one of the three buckets and extracts any explicit parameters. Classification budget: ~500ms. Total prompt-to-render budget: ~4s.

**The visitor cannot tell the layers apart.** Every layer ends in the same place: a real B-Rep solid in the same viewport with the same UI. Do not introduce any visible difference between them.

## 7. Coding conventions

### TypeScript / JavaScript

- TypeScript strict mode is on. No `any` without a `// @ts-expect-error` and a one-line justification.
- File names: `kebab-case.ts` for utilities and modules, `PascalCase.tsx` for React components.
- React components: function components with hooks. No class components.
- Imports: external packages first, then internal aliases (`@/...`), then relative imports. Blank line between groups.
- Async functions: every `await` is wrapped in try/catch or its caller wraps it. No unhandled promise rejections.
- Errors: throw `Error` subclasses with descriptive messages. Never `throw "string"`.
- No `console.log` in committed code outside of explicit logging utilities. Use `console.error` for genuine errors only.
- Component props: typed interfaces named `<ComponentName>Props`, exported alongside the component when other files need them.

### Python

- Python 3.11 syntax. Type hints on every function signature.
- File names: `snake_case.py`.
- Function names: `snake_case`. Class names: `PascalCase`. Constants: `SCREAMING_SNAKE_CASE`.
- Imports: standard library, third-party, local. Blank lines between groups. (Ruff enforces this.)
- All routes return Pydantic models, never raw dicts.
- All exceptions raised from route handlers are `HTTPException` with sensible status codes.
- No `print()` in committed code; use the FastAPI logger.

### Commit conventions

Conventional commits:
- `feat(scope): description` — new functionality
- `fix(scope): description` — bug fix
- `chore(scope): description` — tooling, config, no behavior change
- `docs(scope): description` — documentation changes only
- `refactor(scope): description` — code change with no behavior change
- `test(scope): description` — adding or fixing tests

`scope` is the package or feature area: `web`, `api`, `shared`, `infra`, etc.

One concern per commit. If a single change touches both `web` and `api`, that's two commits. Make this easy by working on one side at a time.

## 8. Hard rules — do not violate without explicit user instruction

1. **Do not beautify the frontend before Phase 9.** Use Tailwind defaults: white background, black text, system font, no custom colors, no transitions, no animations. The visual identity (Royal Blue + white, refined typography, cinematic lighting) is applied in one cohesive pass at Phase 9. Beautifying early creates inconsistency and rework.
2. **Do not add features outside the current phase's scope.** Every phase has a tight deliverables list. If you find yourself building something not in the list, stop and ask.
3. **Do not skip verification checklists.** Each phase ends with a checklist. If any item fails, fix it before declaring the phase complete.
4. **Do not commit secrets.** `OPENAI_API_KEY` and any other secrets live in `.env` (gitignored) and are documented in `.env.example` as empty placeholders.
5. **Do not use the Anthropic API at runtime.** The product calls OpenAI exclusively. Claude Code (you) is a build-time tool only; nothing the user sees at the booth calls Anthropic.
6. **Do not mock data that the spec says must be real.** Geometry must be real B-Rep through Replicad or CadQuery. STEP exports must be real and openable in SolidWorks. The verify panel's face counts and hole counts must be derived from real geometry. Only the verify-panel status messages and the manufacturing-notes filler are scripted/AI-generated; everything else is real.
7. **Do not introduce new dependencies without justification.** The stack in §5 is locked. If a phase legitimately needs a new library, add it to that phase's instruction file and proceed; do not silently add packages.
8. **Do not modify another phase's deliverables.** When working on Phase N, only touch files Phase N owns or that Phase N is explicitly extending. If you find a bug in earlier phase code, fix it in a separate `fix(...)` commit, do not bundle it.
9. **Do not run `git push` unless the user says so.** Commit locally, summarize what changed, wait for the user's go-ahead before pushing.
10. **Do not assume — ask.** If a phase instruction is ambiguous, ask the user before making a guess. Two minutes of clarification beats two hours of rework.

## 9. Frontend aesthetic policy

The demo's visual identity is **Royal Blue and white, sleek, clean, elegant, minimal**. This is applied **only in Phase 9**, after all functional phases are complete and the dress rehearsal passes.

**Phases 0 through 8:**
- Tailwind defaults only.
- No custom colors in `tailwind.config.ts`.
- No custom fonts; use system stack.
- No transitions, no animations, no transforms (except those genuinely needed for 3D viewport behavior).
- White background, black text, plain HTML form controls.
- Spacing is whatever Tailwind utility classes provide.
- Focus exclusively on the demo working correctly. Looks come later.

**Phase 9 (and only Phase 9):**
- Define design tokens in `tailwind.config.ts`: Royal Blue accent `#0056FF` with Azure `#2277FF` as the bright/hover and 3D selection highlight, Soft Lavender `#E3E7FC` tint, white surface, near-black ink. (Earlier docs referenced an ultramarine working value of `#1F3DCC`; superseded.)
- Pick one UI font (refined sans like Geist or Söhne-equivalent) and JetBrains Mono for technical data. Document the choice.
- Apply the cinematic viewport lighting (three-point, soft ground shadow, subtle bloom).
- Apply the full dossier typography pass (blueprint-style dimension lines, 11pt body, A3 title block).
- Single 200ms easing curve for all transitions.

A component built in Phase 2 should look exactly the same as a component built in Phase 7 — equally ugly — until Phase 9 reskins them all together.

## 10. What the demo will NOT do

These are out of scope for the demo. Do not implement them even if they seem easy.

- Generate parts outside the union of Layers 1, 2, and 3. Prompts that fall through Layer 3 with no good match get the closest available result.
- Iterative chat-style refinement. Editing happens via parameter sliders only; re-prompting starts fresh.
- A real verification engine. The verify panel runs simplified checks; the architecture is real, the production-grade verifier is in the internal product build.
- Any physics simulation (FEA, CFD). Mechanism animations are choreographed.
- Real-time multi-user collaboration.
- Drafting-standard drawings (ASME Y14.5, ISO 128). The dossier is an engineering data sheet, not a release drawing.
- Offline operation.
- Mating constraints between separately defined parts. Hero assemblies are pre-assembled at build time.
- Arbitrary user-uploaded CAD file processing (STEP, IGES, DXF). Reference images only (and even those are deferred behind a feature flag).
- Touch device or phone support. Desktop browser only.
